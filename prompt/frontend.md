# 프론트엔드 구현 프롬프트 시퀀스 (모듈 단위 + 검증 게이트)

## 사용 방법

1. 백엔드 Step 0(헬스체크)까지는 최소한 완료되어 있어야 한다 (`GET /health`가 응답해야
   프론트엔드에서 연동 확인이 가능).
2. 아래 프롬프트를 **순서대로** 하나씩 클로드 코드에 입력한다. 한 단계의 "검증 통과"를
   확인하기 전에는 다음 단계로 넘어가지 않는다.
3. 모든 단계는 기능 검증 + **사용성 검증**을 함께 본다. 기능은 동작하지만 사용성이 떨어지면
   (에러 메시지가 불친절하다, 로딩 중인지 알 수 없다 등) 통과로 치지 않는다.
4. 검증 실패/미흡 시 결과를 그대로 붙여넣고 "위 항목을 보완해줘"로 같은 단계를 반복한다.

---

## Step 0. 프론트엔드 초기 셋업 + API 클라이언트 + 기본 레이아웃

```
CLAUDE.md를 읽고 frontend-builder 에이전트를 사용해서 frontend/ 폴더에 Streamlit 앱 뼈대를 구성해줘.

요구사항:
1. frontend/api_client.py
   - 백엔드 4개 모듈(collect, analyze, rag, generate) 엔드포인트를 호출하는 함수들을 모듈화
   - 모든 함수는 타임아웃을 명시하고, 실패 시 예외를 잡아서 사용자가 이해할 수 있는 형태의
     에러 객체/메시지로 변환해서 반환 (원본 트레이스백을 화면에 그대로 노출하지 않는다)
2. frontend/app.py
   - 멀티페이지 구조(st.navigation 또는 pages/ 디렉토리)로 화면을 나눈다:
     ① 앱 선택·수집 ② EDA 대시보드 ③ 감성·토픽 분석 ④ 경쟁사 비교 리포트
   - 사이드바 또는 상단에 "지금 보고 있는 단계가 무엇인지" 항상 알 수 있는 네비게이션 구성
   - 첫 진입 화면에 이 서비스가 무엇을 해주는지 2~3줄 설명 (사용자가 맥락 없이 화면을
     마주치지 않도록)
3. 백엔드 연결 상태를 확인하는 간단한 헬스체크 표시 (예: 사이드바에 "백엔드 연결됨/끊김" 표시)

이 단계에서는 각 화면 내부 콘텐츠는 자리표시자(placeholder)만 두고, 다음 단계부터 실제로 채운다.

구현 후 다음을 직접 실행해서 검증해줘:
1. `streamlit run frontend/app.py`로 정상 기동되는지
2. 백엔드를 끈 상태에서 헬스체크 표시가 "끊김"으로 정확히 뜨는지 (에러로 앱이 죽지 않는지)
3. 네비게이션으로 4개 화면을 모두 오갈 수 있는지
4. 사용성 체크리스트
   - 처음 보는 사람이 "이 서비스가 뭘 해주는지" 3초 안에 알 수 있는 설명이 있는가
   - 현재 어느 화면에 있는지 항상 명확한가

검증 결과를 표로 정리해줘. 모두 통과하기 전까지는 "Step 0 완료"라고 말하지 마.
```

**검증 게이트:** 정상 기동 / 백엔드 끊김 시에도 안정적 표시 / 네비게이션 동작 / 첫인상 설명 존재 — 모두 통과해야 Step 1로 이동.

---

## Step 1. 앱 선택 및 리뷰 수집 트리거 화면

```
frontend-builder 에이전트를 사용해서 "① 앱 선택·수집" 화면을 구현해줘 (api_client.py의
collect 관련 함수를 사용, 직접 requests 호출 금지).

요구사항:
1. 분석 대상 앱(자사 1개 + 경쟁 3~5개) 목록을 선택/확인할 수 있는 UI
2. "리뷰 수집 시작" 버튼 → POST /collect/reviews 호출 → job_id 받아서 진행 상태를 폴링
3. 수집 중에는 진행률/스피너를 보여주고, 사용자가 다른 화면으로 이동했다가 돌아와도
   진행 상태를 다시 확인할 수 있게 한다 (st.session_state 활용)
4. 수집 완료 시 "리뷰 1,000건 이상 수집됨" 같은 성공 기준 대비 결과를 명확히 보여준다
   (미달 시에도 솔직하게 건수를 보여주고 다음 단계 진행 가능 여부를 안내)
5. 이미 수집된 데이터가 있으면 재수집 여부를 사용자에게 먼저 물어본다 (실수로 중복 수집 방지)

구현 후 다음을 직접 실행해서 검증해줘:
1. 정상 케이스: 수집 시작 → 진행 상태 표시 → 완료 후 결과 확인까지 전체 흐름이 끊기지 않는지
2. 실패 케이스: 백엔드가 에러를 반환할 때 사용자가 이해할 수 있는 메시지가 뜨는지
   (스택트레이스 노출 없음)
3. 사용성 체크리스트
   - 버튼을 누른 뒤 "지금 뭔가 진행되고 있다"는 걸 1초 안에 알 수 있는가 (로딩 표시)
   - 완료까지 오래 걸려도 사용자가 멍하니 기다리지 않도록 진행 정보가 갱신되는가
   - 중복 수집 방지 안내가 직관적인가

검증 결과를 표로 정리해줘. 모두 통과하기 전까지는 "Step 1 완료"라고 말하지 마.
```

**검증 게이트:** 정상/실패 흐름 모두 안정적 / 로딩 상태 명확 / 중복 수집 방지 — 모두 통과해야 Step 2로 이동.

---

## Step 2. EDA 대시보드 화면

```
frontend-builder 에이전트를 사용해서 "② EDA 대시보드" 화면을 구현해줘
(api_client.py를 통해 GET /analyze/eda 호출).

요구사항:
1. 앱별 평균 별점, 시기별 리뷰량 추이, 긍부정 비율을 plotly 차트로 시각화
2. 자사 앱과 경쟁 앱을 같은 차트에서 비교할 수 있게 한다 (색상으로 앱 구분)
3. 데이터가 아직 없을 때(Step 1 미완료 상태) 빈 화면 대신 "먼저 리뷰 수집을 진행해주세요"
   안내와 함께 Step 1 화면으로 이동하는 버튼/링크 제공
4. 차트는 마우스오버로 정확한 수치를 확인할 수 있게 한다 (plotly 기본 hover 활용)

구현 후 다음을 직접 실행해서 검증해줘:
1. 실제(또는 샘플) 데이터로 차트가 정상 렌더링되는지
2. 데이터 없는 상태에서 빈 화면 대신 안내 메시지가 뜨는지
3. 앱 간 비교가 색상/범례로 명확히 구분되는지
4. 사용성 체크리스트
   - 차트만 보고도 "어떤 앱이 잘하고 있고 어떤 앱이 문제인지" 한눈에 파악되는가
   - 숫자에 익숙하지 않은 사용자도 범례/축 라벨만으로 이해할 수 있는가 (한글 라벨 사용)

검증 결과를 표로 정리해줘. 모두 통과하기 전까지는 "Step 2 완료"라고 말하지 마.
```

**검증 게이트:** 정상 렌더링 / 빈 상태 안내 / 앱 간 비교 명확성 / 한글 라벨링 — 모두 통과해야 Step 3으로 이동.

---

## Step 3. 감성·토픽 분석 결과 화면

```
frontend-builder 에이전트를 사용해서 "③ 감성·토픽 분석" 화면을 구현해줘
(api_client.py를 통해 GET /analyze/topics, POST /analyze/sentiment, GET /analyze/metrics 호출).

요구사항:
1. 토픽별(로그인, 속도, 혜택 등) 리뷰 비중을 보여주는 차트 + 토픽별 대표 리뷰 예시 노출
2. 감성 분포(긍정/부정/중립) 시각화
3. 분류 모델 성능(F1-score)을 화면 하단 또는 별도 섹션에 "참고 지표"로 표시하되,
   일반 사용자에게는 부담스럽지 않게 접어두는(expander) 형태로 제공
4. 별점-텍스트 불일치 등 한계점을 안내 문구로 함께 노출 (눈에 띄게, 숨기지 않는다)

구현 후 다음을 직접 실행해서 검증해줘:
1. 토픽/감성 결과가 실제(또는 샘플) 데이터 기준으로 정상 표시되는지
2. F1-score 등 기술 지표가 일반 사용자 화면을 어지럽히지 않으면서도 찾을 수 있는 위치에 있는지
3. 한계점 안내가 결과를 신뢰하는 데 방해되지 않으면서도 명확히 보이는지
4. 사용성 체크리스트
   - "가장 시급한 불만 토픽이 무엇인지" 화면 상단만 봐도 알 수 있는가 (중요한 정보 우선 배치)
   - 토픽 대표 리뷰 예시가 있어 추상적 숫자보다 와닿는가

검증 결과를 표로 정리해줘. 모두 통과하기 전까지는 "Step 3 완료"라고 말하지 마.
```

**검증 게이트:** 정상 표시 / 기술지표 비침투적 노출 / 한계점 안내 명확 / 핵심 정보 우선순위 배치 — 모두 통과해야 Step 4로 이동.

---

## Step 4. 경쟁사 비교 리포트 화면 (RAG 기반)

```
frontend-builder 에이전트를 사용해서 "④ 경쟁사 비교 리포트" 화면을 구현해줘
(api_client.py를 통해 POST /generate/report 호출).

요구사항:
1. "리포트 생성" 버튼 → 호출 중에는 최대 30초가량 걸릴 수 있음을 사용자에게 미리 안내
   (예: "리포트 생성에 최대 30초 정도 걸릴 수 있어요" 문구 + 진행 스피너)
2. 생성된 리포트를 읽기 쉬운 구조로 표시 (불만 토픽 → 경쟁사 비교 → 개선 제안 순서 등)
3. 리포트 내 모든 주장에 근거가 된 출처(앱명/문서)를 함께 표시 (RAG 모듈에서 받은 메타데이터 활용)
4. "이 리포트는 공개 자료 기반이며 내부 전략 추정이 아님"을 리포트 상단에 명시
5. 30초를 초과해 응답이 오지 않을 경우를 대비한 타임아웃 처리와 안내 메시지

구현 후 다음을 직접 실행해서 검증해줘:
1. 정상 케이스: 버튼 클릭 → 대기 안내 → 리포트 표시까지 끊김 없이 동작하는지, 실제 소요 시간 측정
2. 타임아웃/실패 케이스: 백엔드 응답 지연 시 사용자가 무한 대기하지 않고 안내를 받는지
3. 리포트의 모든 핵심 주장에 출처가 달려 있는지 직접 확인
4. 사용성 체크리스트
   - 대기 시간 동안 사용자가 불안해하지 않도록 진행 안내가 충분한가
   - 리포트를 처음 보는 사람이 "그래서 우리가 뭘 해야 하는지" 결론 부분만 봐도 알 수 있는가
   - 출처 표시가 신뢰도를 높이면서도 가독성을 해치지 않는가

검증 결과를 표로 정리해줘. 모두 통과하기 전까지는 "Step 4 완료"라고 말하지 마.
```

**검증 게이트:** 정상/타임아웃 흐름 모두 안정적 / 모든 주장에 출처 표시 / 대기 경험 양호 / 결론 가독성 — 모두 통과해야 Step 5로 이동.

---

## Step 5. 전체 UX 통합 점검

```
frontend-builder 에이전트를 사용해서 4개 화면 전체를 사용자 흐름 관점에서 통합 점검해줘.

요구사항 점검:
1. 엔드투엔드 시나리오를 처음 사용하는 사람 입장에서 직접 따라가본다:
   진입 → 서비스 설명 확인 → 앱 선택/수집 → EDA 확인 → 토픽/감성 확인 → 비교 리포트 생성
2. 화면 간 전환 시 맥락이 끊기지 않는지 (예: 리포트 화면에서 갑자기 "데이터가 없습니다"만
   뜨고 어디로 가야 할지 모르는 상황이 없는지)
3. 모든 에러 메시지가 "무엇이 잘못됐고 사용자가 무엇을 하면 되는지"를 담고 있는지 전수 점검
4. 로딩이 필요한 모든 호출(수집, 분석, 리포트 생성)에 일관된 로딩 패턴이 적용되어 있는지
5. 한계점/주의사항 안내 문구가 화면별로 흩어져 있지 않고 일관된 위치/스타일로 노출되는지

발견된 문제는 우선순위(사용자가 막히는 정도 기준 상/중/하)와 함께 목록으로 정리해줘.
이 점검 보고가 끝나기 전까지 "프론트엔드 구현 완료"라고 말하지 마. 상 우선순위 문제가
하나라도 있으면 그것부터 고치고 다시 점검해줘.
```

**검증 게이트:** 엔드투엔드 흐름 끊김 없음 / 에러 메시지 행동 가능성(actionable) / 로딩 패턴 일관성 / 한계점 안내 일관성 / 상 우선순위 문제 0건 — 모두 충족해야 프론트엔드 구현 완료로 간주.
